# ClassAuto 학습 분석 기능 명세 (Analytics Spec)

> 이 문서는 ClassAuto의 **학습 분석 기능**(강의별 + 학기 전체)을 Claude Code로 구현하기 위한 명세서다.
> 6월 현재 목표: **A·B 두 기능 모두**를 가상(synthetic) 데이터 기반으로 구현·검증한다.
> 인프라(이벤트 수집·저장)가 아직 없으므로, **집계·판정·AI 분석 층**을 가짜 데이터로 먼저 완성하고
> 추후 실제 Supabase 집계 결과를 같은 인터페이스에 연결한다.

---

## 0. 핵심 설계 원칙

1. **★ 도메인 범용성 (가장 중요)**: ClassAuto는 **특정 전공에 종속되지 않는 범용 플립러닝 분석 도구**다.
   공학·예술학·법학·자연과학·인문/어학 등 **모든 분야의 교수진**에게 배포된다.
   어떤 코드·프롬프트·UI도 특정 과목(예: 중국어 문법)을 **하드코딩하지 않는다.**
   과목 의존 정보는 전부 **강의 메타데이터로 주입(inject)**한다. (§0-A 참조)
2. **두 시간축 분리**: A(강의별)는 영상이 끝날 때마다 매주, B(학기 전체)는 10주차~학기말에만 작동한다.
3. **판정은 규칙, 설명은 AI**: 혼란/우등/양극화/이탈 4분류는 데이터 기반 규칙으로 판정하고(근거 제시 가능),
   AI는 그 판정을 받아 자연어 브리핑·솔루션을 생성한다. (분류를 AI 감에 맡기지 않는다.)
4. **AI 생성물은 교수자 검토 전제**: 특히 설문 문항·DOI·논문 제안은 반드시 "교수자 검토 필수" 문구를 노출하고,
   교수자가 수정 가능해야 한다. 최종 책임은 교수자에게 있다.
5. **디자인 톤 통일**: A·B 모두 ClassAuto 디자인 시스템 v2(`docs/design-system/`)를 따른다.
   라이트 베이지(`#FAFAF7`) + 골드(`--gold-on-light`) 단일 액센트, 폰트는 `--font-*` 변수로만 참조.
   별도 멀티컬러 팔레트를 새로 만들지 않는다. (§4 참조)

### 0-A. 도메인 추상화 — 과목 메타데이터 주입

특정 과목을 코드에 박지 말고, 강의 생성 시 교수자가 설정하거나 AI가 강의 자료에서 추출한
**`CourseProfile`**을 모든 분석에 주입한다.

```ts
type CourseProfile = {
  subject: string;        // 예: "유체역학", "헌법", "서양미술사", "중국어문법" …
  field: string;          // "공학" | "법학" | "예술학" | "자연과학" | "인문·어학" | …
  // 이 과목에서 학습자가 자주 틀리거나 놓치는 지점(=취약 개념 축). 과목마다 다르다.
  weaknessAxes: string[]; // 어학: ["문법 오류 A","어순"] / 공학: ["오개념","공식 적용 오류"]
                          // 법학: ["쟁점 적용","요건 누락"] / 예술: ["기법","비평 관점 누락"] …
  errorExamples?: string; // (선택) 대표 오류 예시. 프롬프트 품질 향상용. 어학이면 예문 등.
};
```

- 아래 모든 스키마/프롬프트에서 등장하는 "취약 개념"은 `weaknessAxes`로, "과목"은 `subject`/`field`로 치환된다.
- **把字句·보어누락 등은 어디까지나 '어학 전공의 한 예시'일 뿐, 기본값이나 전제가 아니다.**

## 1. 전체 구조

### A. 강의별 개별 분석 (영상 1개 단위 · 매주)
1. 종합 판정 배너 — "이번 영상 종합 결과: X" + 판정 근거
2. 한눈에 보기 4지표 (이해도 / 시청완료율 / 평균 학습시간 / AI Q&A)
3. 진도 분포 — 학생 단위로 일목요연하게 (시청완료율과 통합)
4. 학생별 진척도·취약점 — 시청한 학생 전원 개별화
5. AI 대면 수업 브리핑 — 강의 **전체** 방향
6. 학생별 개인화 솔루션 — **학생별** 교수법 처방

### B. 학기 전체 분석 (10주차 ~ 학기말)
- a. 주차별 학습효율 추이 (ClassAuto 도입 효과)
- b. 학기말 설문 자동생성 (문항마다 교수법 근거 + 참고문헌 DOI, 교수자 검토·수정)
- c. 링크·QR 배포 → 응답 결과 시각화
- d. 학기 총평 [PRO] (교육공학 이론 기반 장점/단점/개선점) → 논문 제목·방향 제안

## 2. A. 강의별 개별 분석

### 2.1 데이터 스키마 (학생 1명)

```ts
type Student = {
  id: number;
  name: string;
  watchedPercent: number;      // 0~1, 시청 진도
  completed: boolean;          // 완주 여부
  dropoffPoint: number | null; // 미완주 시 이탈 지점(%)
  quiz: boolean[];             // 문항별 정/오답 (길이 = 문항 수)
  questionsAsked: number;      // AI Q&A 질문 횟수
  // ★ 오류 축은 CourseProfile.weaknessAxes를 키로 사용 (과목 비종속)
  weaknessHits: Record<string, number>; // { [weaknessAxis]: 빈도 }
  score: number;               // 0~100, 퀴즈 정답률
};
```

> 키를 고정 문자열(보어누락 등)로 박지 말 것. `CourseProfile.weaknessAxes`를 순회해 동적으로 구성한다.

### 2.2 집계 (raw → 지표) — 과목 무관

- `avgScore` 평균 이해도, `completionRate` 완주율, `avgWatched` 평균 시청진도
- `avgQuestions` 인당 평균 질문수, `studyMinPer` 인당 학습시간(환산)
- `stdev` 점수 표준편차 → **양극화 신호**
- `dropBuckets[10]` 이탈 지점 분포(10% 버킷), `dropConcentration` 최대 버킷 집중도(%) → **이탈 신호**
- `perQuestion[]` 문항별 정답률
- `weaknessTotals` = weaknessAxes별 누적 (과목별로 축이 달라짐)
- `progress` { 완료 / 진행중 / 시작만 / 미시청 }
- `roster` 시청 학생별 { level(부진<50 / 보통 / 우수≥75), topWeakness }

> 집계 로직 자체는 어떤 과목이든 동일하다. 바뀌는 것은 weaknessAxes의 내용뿐.

### 2.3 종합 판정 규칙 (4분류 + 근거) — 과목 무관

순수하게 학습 행동 지표(점수·완주·분산·이탈·질문)만으로 판정하므로 **모든 전공에 그대로 적용**된다.

```
if (미완주자 ≥ 전체40% && dropConcentration ≥ 45% && avgScore ≥ 55)
    → "구간 이탈(dropout)"
    근거: 미완주의 N%가 X~Y% 구간 한 곳에 몰림 / 이탈 직전 정답률은 양호 → 영상 구간 의심
else if (stdev ≥ 24 && avgScore ≥ 55)
    → "양극화(polarized)"
    근거: 평균 양호하나 표준편차 큼 / 하위 그룹(<50점) 형성
else if (avgScore < 55 && completionRate < 55)
    → "혼란(confused)"
    근거: 이해도·완주율 동반 저조 / 질문 빈도 높음
else
    → "우등(excelling)"
    근거: 이해도·완주율 동반 양호 / 분산 작음
```

판정별 권장 방향: 혼란→재설명, 우등→심화·문제풀이, 양극화→전체 진도+부진 개별 상담, 이탈→영상 구간 점검.

### 2.4 AI 브리핑 + 학생별 솔루션 — 시스템 프롬프트 (핵심 IP)

집계 결과 + 규칙 판정 + 부진/보통 학생 요약 + 상위 학생 1명 + **CourseProfile**을 입력으로 전달.
프롬프트는 과목명을 **변수로** 받는다.

```
너는 플립러닝 수업 설계를 돕는 교수법 분석 전문가다.
과목: {{subject}} ({{field}} 분야). 이 과목의 학습자 취약 개념 축: {{weaknessAxes}}.
{{#errorExamples}}대표 오류 예시: {{errorExamples}}{{/errorExamples}}

학습자의 사전학습 데이터를 받아, 교수자에게 이번 주 대면 수업 운영안을 제시한다.

판단 원칙(전 분야 공통):
- 이해도·완주율 낮고 질문 많음 → 핵심 개념 재설명
- 이해도·완주율 높음 → 심화·응용·문제풀이
- 평균은 양호하나 소수만 부진 → 전체 진도 + 부진 학생 개별 보충
- 특정 시청 구간에서 다수 이탈 & 그 전 정답률 양호 → 내용이 아니라
  그 구간 영상 자체(길이·난해함·기술 문제)를 점검하라고 지적

취약 개념 분석은 위 {{weaknessAxes}}를 기준으로 하라. (과목에 맞는 용어를 사용)

출력은 JSON만 (코드펜스·여는말 금지):
{
  "verdictSentence": "이번 영상 종합 결과를 교수자에게 알리는 한 문장",
  "briefing": {
    "approachTitle": "...", "approachDetail": "왜+어떻게 2~3문장",
    "openingMove": "대면 첫 5분 실행 지시 한 문장",
    "recommendedMinutes": 60, "focusTopics": ["...1~3개"]
  },
  "studentSolutions": [
    {"name":"", "level":"부진|보통|우수", "weakness":"약점 한 줄", "action":"교수 처방 한 줄"}
  ]
}
studentSolutions는 입력의 부진·보통 학생 전원 + 상위 학생 1명을 포함.
```

> 모델: `claude-sonnet-4-6` 권장 / max_tokens ≈ 1500 / 출력은 `JSON.parse` 전 ```json 펜스 제거.
> `{{subject}}` 등은 실제 호출 시 CourseProfile 값으로 치환. 과목 고정 문구를 프롬프트에 남기지 말 것.

## 3. B. 학기 전체 분석

### 3.1 타임라인 (★중요)

| 시점 | 동작 |
|---|---|
| **10주차** | 교수자에게 공지 배너 노출: "학기 전체 분석 기능이 열렸습니다!" |
| 10주차~ | 강의계획서 입력 → 학습효율 추이 → 설문 자동생성 → 링크·QR 배포 |
| 배포 후 | 설문 응답 수집 |
| **분석 마감 주차** | 응답 집계 → 결과 시각화 → 학기 총평·논문 제안 |

- **분석 마감 주차 = (학기 총 주차 − 1)**
  - 15주 학기 → **14주차**까지
  - 16주 학기 → **15주차**까지
- 학기 길이는 설정값(`semesterWeeks`)으로 받고, 마감 주차를 자동 계산한다.
- 설문 응답 기간 확보를 위해 10주차 배너와 마감 주차 사이에 충분한 간격을 둔다.

### 3.2 플로우 (멀티스텝)

```
[0] 10주차 트리거 배너
 → [1] 강의계획서 입력 (샘플 채우기 제공)
 → [2] (a) 주차별 학습효율 추이 + 10주 종합 요약
 → [3] (b) 설문 자동생성  ※상단에 "교수자 검토 필수" 경고 고정, 문항 편집 가능
 → [4] (c) 링크·QR 배포
 → [5] (c) 응답 결과 시각화
 → [6] (d) 학기 총평[PRO] + 논문 제목·방향
```

### 3.3 a. 주차별 학습효율 추이

- 1주~현재까지 누적 지표(완주율 / 평균이해도 / 대면참여도 등)의 라인 차트.
- 1주 대비 상승폭(+%p)을 강조해 ClassAuto 도입 효과를 가시화.
- 지표는 학습 행동 기반이라 과목 무관.

### 3.4 b. 학기말 설문 자동생성

- **상단 고정 경고**: "⚠️ AI를 통한 생성물은 반드시 교수자 검토가 필요합니다. … 최종 책임은 교수자에게 있으며, 특히 DOI는 실재 여부를 확인하세요."
- 문항 6개 (5점 리커트 5 + 주관식 1).
- **문항마다 반드시**: ① 본문(편집 가능 textarea) ② 교수법 설계 근거 ③ 참고문헌 citation + DOI.
- 이 수업의 실제 약점(CourseProfile.weaknessAxes)을 측정하는 문항 포함. 인식·자기효능감·전이·만족도 균형.

#### 설문 생성 시스템 프롬프트 (핵심 IP) — 과목 변수화

```
너는 교육공학 전문가다. 과목 {{subject}} ({{field}} 분야)의 10주차까지 학습 데이터와
강의계획서를 받아 학기말 학습자 설문을 설계한다.
- 문항 6개: 5점 리커트 5 + 주관식 1.
- 각 문항마다 (1) 교수법적 설계 근거와 (2) 실제 교육공학 문헌(citation + DOI)을 제시.
- 이 수업의 실제 취약 개념({{weaknessAxes}})에 대한 자기효능감을 측정하는 문항 포함.
- 인식·자기효능감·전이(transfer)·만족도를 균형 있게.
출력은 JSON만:
{ "questions": [ {
  "no":1, "text":"", "scale":"5점 리커트|주관식",
  "rationale":"교수법 근거 1~2문장",
  "reference": { "citation":"저자(연도). 제목. 출처.", "doi":"10.xxxx/xxxxx" }
} ] }
```

> 교육공학 설문 이론(자기효능감·전이·만족도·형성평가 등)은 전 분야 공통이며, `{{weaknessAxes}}`만 과목별로 달라진다.

### 3.5 c. 배포 → 결과 시각화

- 설문 링크 + QR 생성·배포 (수강생 N명).
  - 데모는 가짜 QR(SVG 패턴). **실제 구현 시 `qrcode` 라이브러리로 교체.**
- 응답 수집 후: 문항별 5점 척도 분포(1=빨강 … 5=초록) + 평균 시각화.

### 3.6 d. 학기 총평 [PRO] + 논문

- **PRO 기능**(배지 표시).
- 교육공학 이론 렌즈 명시 + 장점/단점/개선점.
- **논문은 제목·방향까지만** 제안 (제목 / 데이터 기반 주장 방향 / 권장 연구방법).
  - 실제 집필은 별도 서비스 **Accept.best**로 연결.

#### 총평 생성 시스템 프롬프트 (핵심 IP) — 과목 변수화

```
너는 교육공학 연구자다. 과목 {{subject}} ({{field}})의 학습 데이터·강의계획서·학기말 설문 결과를
종합해 교수자를 위한 학술적 학기 총평을 작성한다. 최신 교육공학 이론(플립러닝, 인지부하,
멀티미디어 학습, 자기조절학습, 형성평가 등)에 근거하라.
출력은 JSON만:
{
  "overview":"종합 총평 2~3문장",
  "theoryLens":"적용한 핵심 이론 1~2개 한 줄",
  "strengths":[], "weaknesses":[], "improvements":[],
  "paperSuggestions":[ {"title":"국문 제목","direction":"데이터로 무엇을 주장할지","method":"권장 방법"} ]
}
paperSuggestions는 2개. 이 수업의 실제 데이터로 쓸 수 있는 현실적 주제로(해당 전공 맥락 반영).
```

### 3.7 할루시네이션 / DOI 처리

- 현재 프롬프트는 AI에게 DOI를 직접 쓰게 하므로 **가짜 DOI가 생성될 위험이 높다.**
- 단기: "교수자 검토 필수" 경고 + 교수자 편집으로 방어.
- **정식 구현 시: Crossref 또는 Semantic Scholar API로 실재 논문을 검색해 DOI를 주입**하는 구조로 교체.
  (B는 학기말 기능이라 6월 우선순위 아님 — 가짜 데이터 검증 단계에서는 경고 문구로 충분.)

## 4. 디자인 시스템

> **원칙: ClassAuto 디자인 시스템 v2(`docs/design-system/`)를 그대로 따른다.**
> 이 명세의 분석 화면은 별도 색을 새로 만들지 않고 v2 토큰만 사용한다.
> (CLAUDE.md v2와 충돌 금지 — 아래는 v2 정책의 요약이며, 항상 `docs/design-system/`이 최종 기준.)

### 표면·색 (v2 — 라이트 베이지 + 골드)
- 분석 화면(교수자 측)은 **라이트 베이지 표면** (`#FAFAF7`) 기준. 카드 흰색.
  - 다크 표면(`#0A0A0A`)은 학생 영상 player·인터스티셜 퀴즈 한정 — 분석 화면에는 쓰지 않는다.
- 액센트/PRO 강조: **골드** — 라이트 위 `--gold-on-light` (`#B88308`), 다크 위 `--gold` (`#FFB627`).
- ❌ v1의 violet/cyan/pink·멀티컬러 그라데이션 사용 금지(폐기됨). 분석 화면도 예외 아님.

### 차트의 의미색 (유일하게 허용되는 컬러)
- CLAUDE.md v2가 "의미적 컬러(빨강·녹색)는 **교수자 차트** + 가벼운 UI 인디케이터 한정"으로 허용.
- 분석 차트는 이 범위 안에서만 색을 쓴다:
  - 상태/등급: 부진=red, 보통=amber, 우수=green (의미 전달용, 장식 아님)
  - 5점 척도 분포: 1점(red) → 5점(green) 그라데이션 (의미 매핑)
- 그 외 데이터 계열 구분이 필요하면 **골드 단일 톤의 명도/투명도 차이**로 처리(보라·핑크 등 도입 금지).

### 폰트 (변수로만 참조)
- `--font-body`(본문·UI·숫자, Pretendard `tabular-nums`) / `--font-display`(헤드라인, Paperlogy) / `--font-han`(한자 강조).
- 폰트명을 코드에 직접 박지 말 것. Geist 등 타 폰트 금지.
- 과목이 한자를 포함할 때만 `.han` 클래스(`--font-han` + `--gold-on-light`) 사용. **범용 도구이므로 한자 강조를 기본값으로 깔지 말 것.**

### 컴포넌트·기타
- 둥근 카드(radius 12~14), 도넛 진도차트, 히트맵 셀, 학생 카드 그리드.
- Easing은 `--ease-out`/`--ease-spring`, 진입 fade-in stagger(80ms), `prefers-reduced-motion` 지원, **localStorage 금지**(React state/서버 세션).
- ※ 특정 과목(언어·예술 등)을 연상시키는 장식·색을 기본값으로 쓰지 말 것. 중립 유지.

> 프로토타입(`*.jsx`)의 인라인 멀티컬러(`#3B6FB6`·purple·pink 등)는 시연용 임시값이다.
> 정식 구현 시 위 v2 토큰으로 교체하며, 그 색들을 코드에 남기지 않는다.
