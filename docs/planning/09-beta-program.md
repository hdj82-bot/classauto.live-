## 1. 가치 명제 재정의 (중요)

ClassAuto의 진정한 가치는 **불쾌한 골짜기를 줄인 아바타**가 아니라:

> **동적 영상 → 학습자가 영상에 자유롭게 질문하고 퀴즈를 풀며 상호작용 → 그 데이터를 모아 교수자가 차주 대면 수업의 방향을 결정하는 플립러닝 루프.**

따라서 베타·논문·제품의 중심축은 **상호작용 데이터 루프의 계측**이다. 아바타(인트로/아웃트로/Q&A)는 부차적 변수로 둔다. 이 재정의가 베타 설계 전반을 지배한다.

## 2. 연구 설계

### 2.1 목표 저널 (먼저 정한다)

전공(중국어 교수법·플립러닝)에 맞춰 **CALL 계열 우선**:

- *Computer Assisted Language Learning (CALL)*, *ReCALL*, *Language Learning & Technology* — 언어 학습 + 테크 (best fit)
- *Computers & Education*, *British Journal of Educational Technology (BJET)*, *Education and Information Technologies* — 일반 EdTech (IF 높음)

투고 대상의 기대(표본·도구·윤리)를 빌드 전에 확인하고 그에 맞춰 측정 설계.

### 2.2 연구 프레이밍

깨끗한 RCT보다 **DBR(설계기반연구) + 학습 분석(learning analytics)**이 현실적이고 강하다. "상호작용 데이터가 취약 개념을 드러내고, 그것이 교수자의 대면 수업을 바꾼다"는 루프가 곧 논문 thesis이자 제품 가치 명제.

### 2.3 연구 질문 (초안)

| RQ | 내용 | 유형 |
|---|---|---|
| RQ1 | 플립 중국어 수업에서 학생은 인-비디오 Q&A·퀴즈와 어떻게 상호작용하는가 | 기술적 학습 분석 |
| **RQ2 (핵심)** | 집계 상호작용 데이터(질문·퀴즈 오답·재시청 히트맵)가 학생 취약점을 신뢰성 있게 드러내고, 교수자의 대면 수업 초점과 정렬/개선되는가 | DBR |
| RQ3 | 상호작용형 영상이 수동 시청 대비 참여·완주·성취와 관련되는가 | 준실험 |
| RQ4 (부차/Future Work) | 아바타 답변이 텍스트 대비 인지된 지원감·만족도에 영향을 주는가 | UX 소실험 |

> 아바타(RQ4)는 소표본·교차교란으로 인과 "증명"이 어렵다. 논문 중심은 RQ2에 둔다.

## 3. 측정·로깅 스키마 (최우선 — 소급 수집 불가)

**코드보다 먼저** 연구 데이터 스키마를 확정한다. 9월에 못 모은 데이터는 영원히 없다.

| 범주 | 수집 항목 |
|---|---|
| **질문 이벤트** | 질문 텍스트, 타임스탬프, 영상 내 위치(초), RAG 유사도 점수, 거부 여부, 아바타 요청 여부, 클러스터 ID |
| **퀴즈** | 시도·정답·소요시간, 문항별 오답 패턴, 인터스티셜 응답 지연 |
| **시청** | play/pause/seek, 구간(슬라이드)별 체류, 재시청 횟수, 완주율, 딴짓 경고 발생 |
| **교수자 루프** | 대시보드 열람, 데이터 기반 *실제 행동*(차주 수업 변경) — 인앱 로그 + 인터뷰 |
| **성과** | 퀴즈/사전·사후 테스트 점수, (가능 시) 기말·성적 상관 |
| **인식(검증 척도)** | 동기 ARCS/IMMS, 인지부하 Paas, 유용성 TAM, (아바타) 사회적 실재감 척도 |

> 슬라이드쇼 재생(§5, `08`) 구조 덕에 **슬라이드별 체류·재시청 히트맵이 네이티브로 수집**된다(이미 Pro 기능 `01-pricing-policy.md:158`). 논문용 자산.

> **구현 가능한 데이터 모델은 [10-research-data-model.md](./10-research-data-model.md) 참조** — 기존 스키마 재사용 분석 + 신규/확장 테이블(컬럼·타입·인덱스·마이그레이션 순서). 빌드 전 확정 필수.

## 4. 코호트·모집·운영

- **교수자 5~15명** — 양보다 질. 실제로 쓰고 피드백 줄 동기 있는 분. 저자 본인 중국어 강좌 포함(insider DBR). 분야 stratify(어학 vs 비어학).
- **분석 포함 최소 기준**: 강좌당 ≥N강의, ≥M명 학생 (확정 필요)
- **화이트글러브 온보딩**: 첫 PPT 업로드 → 베이스 얼굴 1회 녹화 → 첫 강의 발행까지 직접 지원
- **주간 피드백 루프**: 오피스아워 + 인앱 피드백 위젯 + 베타 커뮤니티
- 베타 교수자는 **Pro 무료**(`01-pricing-policy.md:227`)

## 5. 영상·아바타 운영 모델 (A/B)

교수자 1명 / 월 기준:

| 항목 | 값 |
|---|---|
| plain 강의(채팅 Q&A만) | **하드캡 없음** — 이제 저원가, 검정력 위해 넉넉히 |
| 아바타 Q&A 활성 강의 | **월 2편** |
| 아바타 Q&A 영상당 렌더 | **영상 전체에서 3렌더** (학생당 아님) |
| 렌더 대상 | **야간 배치로 상위 3개 질문 클러스터** — 선착순 아님 |
| 즉시 응답 | 항상 RAG 텍스트(무료). 아바타는 다음날부터 캐시로 즉시 제공 |
| 과금 | 월 정액, **크레딧 차감 없음** |
| 비교·수렴 | 2026년 9월 학기 종료 후 |

### 5.1 Q&A 지연(latency) 대응

HeyGen 렌더는 수 분 소요 → "탭하면 즉시 아바타"는 캐시 적중/사전 렌더일 때만 가능. 모델:

```
학생 질문 → RAG 텍스트 답변(즉시·무료)
          → 캐시 미스: 텍스트만 + 질문을 클러스터링 큐에 적립
          → 야간 배치: 상위 3개 클러스터만 아바타 렌더
          → 다음날 유사 질문: 캐시에서 아바타 즉시 제공
```

비용은 교수자당 월 6렌더(2영상 × 3)로 고정. 가장 많이 겹친 질문에 렌더가 쓰여 가치 극대화.

### 5.2 투명성

캐시 아바타 답변은 원 질문을 함께 표시("비슷한 질문에 대한 답변입니다"). **권위 있는 답은 학생 질문에 맞춰 생성된 텍스트(Claude RAG)**, 아바타는 전달 방식의 부가.

## 6. A/B / 비교 설계 (방향성 신호)

- 채팅 강의 vs 아바타 강의는 **서로 다른 강의**라 교란(내용·난이도·시점) → 인과 "증명" 아님, **정성·방향성** 신호.
- 표본 소수 → 통계 결론 무리. 효과크기·신뢰구간으로 보고.
- **사전 정의 지표**(빌드 전 확정): 완주율, 퀴즈 점수, "이 답변 도움됐나요?" 1클릭 설문, 주간 활성 학습자.
- 더 깨끗이: 같은 영상 안 학생 무작위 배정 — 베타엔 과함(향후).

## 8. ⚠️ IRB · 윤리 · 동의 (하드 데드라인)

- 학생 데이터로 논문을 쓰려면 **9월 전 기관생명윤리위(IRB) 승인 + 사전 동의** 필수. 여름에 처리 안 하면 데이터를 모아도 게재 불가.
- **연구용 동의(옵트인·익명화)**를 제품의 "졸업 후 자동 삭제"(`CLAUDE.md`) 약속과 **분리** 설계.
- 동의 거부 학생도 시청은 가능하되 분석에서 제외.

## 10. 성공 지표 (상용화 판단)

| 축 | 지표 |
|---|---|
| Activation | 교수자가 첫 강의를 발행한 비율 |
| Engagement | 강의당 학습자 수, 학생당 질문 수, 완주율 |
| Retention | 주간 활성 학습자 추이 |
| Satisfaction | 설문/NPS, "도움됐나요?" |
| **핵심 루프** | 교수자가 대시보드 데이터로 **실제 대면 수업을 바꾼 횟수** (제품 가치 + RQ2 발견) |

이 지표가 상용화 go/no-go와 논문 결론을 동시에 뒷받침한다.
